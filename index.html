Developer workflow

AI CLI Tools

Why CLI?

  • Close to the repo: run in your project context
  • Composable: scripts + pipes + CI-friendly outputs
  • Fast loops: generate, test, refine without leaving the terminal

Popular examples

  • Codex (CLI): agentic coding and refactors from the terminal
  • Gemini (CLI): prompt/assist workflows integrated with developer tooling
  • Claude Code: codebase-aware coding assistance via terminal workflows
  • Also common: linters/test runners + "AI wrappers" around them

Treat CLIs as teammates: give tasks, constraints, and insist on diffs + tests.

Safety & quality

CLI Guardrails

Do

  • Work in small steps: one change-set at a time
  • Require evidence: tests/lints/build output
  • Ask for a plan before large refactors or migrations
  • Review diffs like any PR (security, logic, edge cases)

Avoid

  • Secrets in prompts: API keys, tokens, client data
  • Unbounded permissions: tools that can delete/modify prod without checks
  • Blind merges: "looks good" without running tests
  • Scope creep: agent changing unrelated files
Least privilege Human review No secrets

If you can't explain what changed and why, don't ship it.

Practical prompting

Prompt Engineering

Prompt engineering means giving the model clear intent, context, constraints, and expected output format.

Bad prompt

"Write test cases for my API."

  • Too vague: no endpoint details, auth, edge cases, or output format.
  • Results are often generic and less useful.

Good prompt

"You are a QA engineer. Create 15 API test cases for POST /users with fields name, email, and age. Include positive, negative, boundary, and security cases. Return as a table with columns: id, scenario, request, expected status, expected response."

  • Provides role, scope, data, constraints, and response structure.
  • Output is actionable and ready for execution.
Practical prompting

Context Engineering

Prompt engineering writes the instruction. Context engineering supplies the right information at the right time.
